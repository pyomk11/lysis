# AI 협업 개발 일지

> Lysis 프로젝트에서 AI 도구를 활용한 핵심 개발 사례를 정리한 문서입니다.
> 단순히 "AI를 썼다"가 아니라, **어떤 문제를 어떤 방식으로 AI와 함께 해결했는지** 기록합니다.

## 사용한 AI 도구 요약

| 도구 | 역할 | 활용 단계 |
|------|------|---------|
| Claude Code | 주 개발 에이전트 — 코드 작성, 리팩터링, 디버깅, 배포 | 전 개발 과정 |
| Claude Cowork | 문서 작성 보조 — AI 리포트, 발표 자료 | 문서화 단계 |
| ChatGPT | 레드팀 — 프롬프트 우회 테스트, 보안 검증 | 프롬프트 설계 |

## 사례 1: RLS 무한 재귀 버그 해결

### 상황
교수자 로그인 기능을 구현한 후 로그인이 작동하지 않았습니다. 로그인 버튼을 누르면 "로딩중..."이 잠깐 뜨고 다시 로그인 화면으로 돌아오는 현상이 반복되었습니다.

### AI와의 협업 과정
1. **증상 전달**: Claude Code에 "로그인이 안 돼. 로그인 정보 입력하고 로그인 누르면 로딩중.. 이라고 잠깐 뜨고 다시 로그인 창으로 돌아와"라고 전달
2. **1차 분석**: Claude Code가 코드 흐름을 추적 — `signIn()` → `getProfile()` → null 반환 → `/login`으로 리다이렉트되는 루프 발견
3. **원인 진단**: `getProfile()`이 null을 반환하는 이유를 Supabase RLS 정책에서 찾음. `teachers_see_students`라는 정책이 profiles 테이블을 참조하면서 자기 자신을 다시 조회하는 **무한 재귀**가 발생
4. **해결**: 문제가 되는 RLS 정책을 `DROP POLICY`로 제거하고, `profiles_select_own` 정책을 `auth.uid() = id` 조건으로 단순화

### 배운 점
- Supabase RLS에서 "다른 테이블을 참조하는 정책"은 재귀 위험이 있음
- 로그인 루프는 겉으로 보면 프론트엔드 문제 같지만, 실제 원인은 DB 정책일 수 있음
- Claude Code가 프론트엔드 → API → DB까지 전체 스택을 한 번에 추적할 수 있어서 빠르게 원인을 찾음

## 사례 2: 3-에이전트 파이프라인 설계

### 상황
초기에는 하나의 Gemini 호출로 가드레일 + 진단 + 응답을 모두 처리했습니다. 그러나 "답만 알려줘"를 차단하면서 동시에 유도 질문을 생성하는 것이 단일 프롬프트로는 불안정했습니다.

### AI와의 협업 과정
1. **문제 정의**: Claude Code에 "가드레일이 가끔 뚫리고, 진단 내용이 학생에게 직접 노출되는 경우가 있다"고 설명
2. **아키텍처 제안**: Claude Code가 3-에이전트 파이프라인 구조를 제안 — 각 에이전트가 독립적으로 하나의 역할만 수행
3. **구현**: `route.ts`에 `runGuardrail()`, `runDiagnosis()`, `runSocratic()` 세 함수를 분리하여 순차 호출
4. **레드팀**: ChatGPT에 "이 가드레일 프롬프트를 우회할 방법 10가지를 알려줘"라고 요청 → 발견된 우회 패턴을 GUARDRAIL_PROMPT의 차단 기준에 추가
5. **반복 개선**: Claude Code로 프롬프트를 수정하고, ChatGPT로 다시 우회 테스트 → 3회 반복

### 설계 결정
- Agent B(진단)의 출력을 JSON으로 강제하여 자유 텍스트 답변 유출을 구조적으로 차단
- Agent A(가드레일) 실패 시 `allowed: true`로 폴백 — 학습 중단보다 간헐적 우회가 덜 해로움
- Agent B → Agent C 사이에 진단 JSON만 전달, 원본 에러 메시지 직접 전달 금지

## 사례 3: 교수 대시보드 — 교수별 데이터 격리

### 상황
교수 대시보드가 전체 강의의 데이터를 보여주고 있었습니다. 여러 교수가 사용하려면 자기 강의만 볼 수 있어야 합니다.

### AI와의 협업 과정
1. **요구사항 전달**: "교수자로 로그인 했을 때 전체 강의 대시보드가 뜨는데, 자기 강의만 확인할 수 있도록 수정해줘"
2. **Claude Code 대응**:
   - `classes` 테이블에 `teacher_id` 컬럼 추가하는 마이그레이션 SQL 생성
   - API(`/api/dashboard/route.ts`)에서 `teacherId` 파라미터를 받아 해당 교수의 반만 필터링하도록 수정
   - 프론트엔드에서 `profile.id`를 `teacherId`로 전달하도록 수정
3. **추가 요구**: "기간을 원하는 범위로 볼 수 있게 바꿔줘" → 날짜 범위 선택 UI + API 필터링 추가

### 배운 점
- 데이터 격리는 API 레벨(쿼리 필터)과 DB 레벨(RLS) 양쪽에서 모두 처리해야 안전
- Claude Code가 프론트엔드 → API → DB 스키마까지 한 번에 수정 — 수동으로 했으면 파일 3~4개를 일일이 동기화해야 했음

## 사례 4: 모바일 반응형 전면 수정

### 상황
/app 페이지(에디터 + 채팅)가 모바일에서 에디터와 채팅이 동시에 표시되어 사용 불가능했습니다. 랜딩 페이지, 대시보드도 모바일 대응이 안 되어있었습니다.

### AI와의 협업 과정
1. **요구사항**: "모바일 부분 전반적으로 수정해보자"
2. **Claude Code가 수행한 작업들**:
   - `/app`: `mobileTab` 상태 추가 → 모바일에서 에디터/채팅을 탭으로 전환하는 UI 구현
   - 에디터/채팅 패널에 `hidden md:flex` / `md:block` Tailwind 클래스 적용
   - 채팅 탭에 읽지 않은 메시지 배지 추가
   - 랜딩 페이지: `landing.css`에 모바일 breakpoint 추가 (nav 56px, hero padding, footer flex-direction)
   - 대시보드: 헤더 스택 방향 변경, 날짜 입력 크기 조정, 반 선택기 full-width
3. **결과**: 한 번의 대화로 3개 페이지의 모바일 대응을 일괄 처리

### 배운 점
- "전반적으로 수정"이라는 모호한 요청에도 Claude Code가 각 페이지의 문제점을 식별하고 적절한 해결책 적용
- Tailwind의 반응형 유틸리티(`md:`, `hidden`)를 활용하면 별도 CSS 없이 깔끔하게 처리 가능

## 사례 5: 게스트 모드 설계

### 상황
로그인 없이도 Lysis를 체험할 수 있어야 한다는 요구사항이 추가되었습니다. 그러나 비로그인 사용자의 데이터가 DB에 저장되면 안 됩니다.

### AI와의 협업 과정
1. **요구사항**: "로그인 없이 체험만 할 수 있게도 해줘. 로그인 없이 하면 DB에 저장은 안 되게."
2. **Claude Code 설계**:
   - URL 파라미터 기반: `/app?guest=true`로 게스트 모드 진입
   - 로그인 페이지에 "로그인 없이 체험하기" 링크 추가 → `/app?guest=true`로 이동
   - `/app` auth 체크에서: guest=true면 초대코드 스킵 + 세션 미생성
   - `createSession()`, `saveMessage()` 등 모든 DB 저장 로직에 `if (profile)` 조건 추가
3. **추가 수정**: "수업 초대코드는 로그인을 한 상태에서만 뜨도록 수정해줘" → 비로그인 시 초대코드 화면 스킵

### 배운 점
- 게스트 모드는 "기능을 추가"하는 것이 아니라 "기존 기능에서 조건부로 빼는 것"
- AI에게 명확한 제약 조건("DB 저장 안 됨")을 전달하면 일관성 있는 구현이 나옴

## 사례 6: 랜딩 페이지 일관성 검토

### 상황
개발이 진행되면서 랜딩 페이지의 설명이 실제 구현과 맞지 않는 부분이 생겼습니다.

### AI와의 협업 과정
1. **요청**: "랜딩 페이지에 현재 프로젝트랑 모순되는 내용이 있는지 확인해주고, 있으면 바로 바꾸지 말고 어떻게 바꾸는 게 좋을지 말해줘."
2. **Claude Code 분석 결과**:
   - "URL 하나면 지금 바로 수업에서 쓸 수 있습니다" → 로그인 시스템이 생겼으므로 부정확
   - "나만의 학습 기록" 기능 설명 → /progress 페이지 내용과 불일치
   - 예시 통계 "아! 순간 기록 수 58회" → 실제 대시보드에 없는 메트릭
   - "LMS나 과제 시스템 없이도" → 로그인 시스템 존재와 어색
3. **수정 방향을 먼저 논의**한 뒤 승인 후 수정 진행

### 배운 점
- AI에게 "확인만 해줘, 바로 바꾸지 마"라고 지시하면 검토→논의→수정 3단계로 나뉘어 실수를 줄일 수 있음
- 기획과 구현이 분리되면서 생기는 문서 불일치는 개발 후반에 반드시 체크해야 함

## 사례 7: AI 리포트 작성 (Claude Cowork)

### 상황
공모전 제출용 AI 빌딩 리포트의 내용이 초기 기획 기준으로 작성되어 있어, 실제 구현과 많은 부분이 달랐습니다.

### AI와의 협업 과정
1. **Claude Cowork에 기존 리포트 + 프로젝트 코드 분석 요청**
2. **Cowork가 발견한 불일치**:
   - 런타임 모델: Claude Sonnet/Haiku로 기재됨 → 실제는 Gemini 2.5 Flash/Flash Lite
   - DB: Vercel Postgres로 기재됨 → 실제는 Supabase
   - 미구현 기능(프롬프트 캐싱, CI eval 스크립트, ast 폴백 등)이 기재됨
3. **Cowork가 docx 스킬을 활용하여** 원본 양식을 유지하면서 내용을 실제 구현에 맞게 재작성
4. **문구 협업**: "URL 하나면" 문구 수정 시 3가지 옵션을 제안받고 선택하는 방식으로 진행

### 배운 점
- 코드를 직접 분석하는 AI에게 리포트를 검증받으면 "기획서 vs 실제 구현"의 괴리를 빠르게 잡을 수 있음
- Cowork의 docx 생성 기능으로 서식이 있는 문서도 코드 기반으로 생성 가능

## AI 협업 원칙 (개발 과정에서 체득)

1. **명확한 제약 조건 전달**: "DB 저장 안 됨", "바로 바꾸지 마"처럼 경계를 명시하면 AI 출력 품질이 올라감
2. **레드팀은 다른 모델로**: Claude Code가 만든 프롬프트를 ChatGPT로 공격 → 같은 모델은 자기 취약점을 잘 못 찾음
3. **전체 스택 맥락 공유**: 프론트→API→DB를 한 번에 설명하면 AI가 일관성 있는 수정을 해줌. 파편적으로 요청하면 동기화가 깨짐
4. **검토→논의→수정 3단계**: 특히 문서·UI 카피 수정은 "확인만 해줘"로 먼저 검토 후 방향을 논의하는 것이 안전
5. **CLAUDE.md 유지**: 프로젝트 컨벤션 파일을 최신 상태로 유지하면 AI가 맥락을 잃지 않음
